# UX-аудит Support Communication, 2026-07-01

## Область

Проверены текущие экраны локального приложения `http://127.0.0.1:5173/`: публичный лендинг, вход, валидация входа, onboarding организации, рабочее место диалогов, настройки и администрирование сервиса. Скриншоты сохранены в этой же папке.

## Шаги и состояние

1. Публичный лендинг - в целом здоровый. Ценность продукта понятна, но CTA конкурируют между собой: `Trial`, `Запустить trial`, `Получить демо`, `Уже есть аккаунт`.
2. Экран входа - частично здоровый. Есть понятные режимы Login/SSO/Invite/Blocked/Expired/Maintenance, но диагностические режимы стоят рядом с пользовательским входом.
3. Ошибка входа - проблемный. Пустой email дает понятную ошибку, но валидно выглядящие данные возвращают сырой `Internal Server Error`.
4. Onboarding - проблемный. Есть чеклист и сводка, но прогресс не объясняет, какой пункт не засчитан; после тестового сообщения остается блокировка `5 из 6`.
5. Рабочее место диалогов - критично проблемный. При пустой очереди центральная зона продолжает показывать статус, handoff summary, composer и карточку клиента, что выглядит как выбранный диалог.
6. Настройки - проблемный. Экран перегружен: матрица ролей, лимиты, каналы, SDK, webhooks, security и backend diagnostics находятся в одном длинном рабочем пространстве.
7. Админ сервиса - частично здоровый. Опасные действия защищены причиной и подтверждением, но контекст платформы, организации и изменения статуса слишком плотно совмещен.

## Приоритетные правки

### P0 - исправить состояния, которые ведут к ошибочным действиям

1. Разделить `empty`, `error` и `selected conversation` в диалогах. Если диалогов нет или API упал, центральная зона не должна показывать handoff summary, статус, composer и карточку клиента. Покажите полноэкранное состояние: причина, что уже загружено, кнопка повтора, ссылка на статус API.
2. Заменить `Internal Server Error` во входе на пользовательское сообщение с кодом трассировки: "Не удалось проверить учетные данные. Попробуйте позже или войдите через SSO." Сырой backend-текст оставить только в деталях для поддержки.
3. Починить onboarding-гейтинг. Чеклист должен показывать незавершенный пункт, конкретное поле и действие. Если `Лимиты` засчитаны по умолчанию, прогресс должен стать `6/6`; если нет, у шага нужен явный `Сохранить лимиты`.

### P1 - снизить когнитивную нагрузку

4. Упростить маршруты на лендинге: один основной CTA `Запустить trial`, вторичный `Войти`, демо вынести ниже или в тарифы. Сейчас пользователь видит несколько похожих выходов из первого экрана.
5. В настройках разбить экран на рабочие зоны: `Роли и доступ`, `Каналы`, `SDK и webhooks`, `Security`, `Backend diagnostics`. На первом экране оставить только статус, ошибки и быстрый переход к проблемным областям.
6. В service admin отделить мониторинг платформы от операций с организациями. Для privileged action держать sticky-панель подтверждения рядом с выбранной сущностью, а не ниже длинного списка.

### P2 - доступность и читаемость

7. Исправить матрицу ролей в настройках: сейчас строки визуально слипаются (`РольПанельНастройкиПаролиОтчеты`). Нужна настоящая таблица с заголовками колонок, границами и читаемым порядком фокуса.
8. Увеличить различимость disabled/secondary nav items. В service admin отключенная навигация выглядит почти как обычная, хотя имеет важное ограничение доступа.
9. Проверить контраст красных/зеленых бейджей и маленьких статусов, особенно `работает`, `деградация`, `ошибки`, `Новый`. Со скриншота видно, что часть статусов держится на цвете и мелком тексте.

## Ограничения проверки

Проверка основана на свежих скриншотах и DOM локального приложения. Полное WCAG-соответствие не подтверждалось: нужны отдельные проверки клавиатурной навигации, screen reader labels, zoom/reflow и контраста инструментами.

Для открытия защищенного рабочего экрана была временно задана демо-сессия в `sessionStorage` текущей вкладки браузера; файлы проекта не изменялись.
