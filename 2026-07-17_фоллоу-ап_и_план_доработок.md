# Follow-up встречи по системе КС

## Короткий итог

Система выглядит перспективной и может закрыть ключевые потребности поддержки. Критичный для теста блок — автоматическое перераспределение обращений из ожидания при появлении свободного оператора. Остальные изменения предлагается вести по приоритетам ниже.

## Договорённости и решения

- Номер телефона передаётся из текущей системы, если он доступен в канале; если нет — оператор указывает его вручную. Telegram ID не заменяет номер телефона.
- Нужна проверка возможности получать и показывать геолокацию клиента; это полезно для диагностики VPN. Информация об устройстве (iOS/Android) желательна, но не критична.
- Внутренние комментарии следует показывать непосредственно в ленте диалога, визуально отделяя от сообщений клиента и оператора.
- Для тематик в чатах нужен быстрый поиск по вводу, а не только выпадающий список.
- Оценка клиентом должна оставаться цифровой. После любой оценки клиенту нужно предлагать оставить комментарий; комментарий не должен создавать новое обращение и должен быть виден в истории диалога.
- На панели нужны: активные диалоги, ожидающие ответа, статусы операторов и число текущих диалогов у каждого оператора. Панель должна оставаться компактной; нужен отбор/сортировка по онлайн-операторам.
- В карточке клиента история обращений должна открываться для просмотра; достаточно модального окна с датой и содержимым обращения.
- В шаблонах нужен поиск/автоподбор прямо при наборе текста в чате: по ключевому слову появляется подсказка, выбор по Enter вставляет шаблон. Отдельная страница шаблонов должна поддерживать поиск, редактирование и создание.
- Маршрутизация должна равномерно распределять новые и ожидающие обращения между доступными операторами с учётом лимитов. При освобождении слота обращение из ожидания должно назначаться автоматически.
- Операторы входят в систему офлайн и вручную меняют статус; этого достаточно, чтобы администраторы не получали чаты. Доработка не требуется.
- Статус «Завершает диалоги» нужен перед перерывом/обедом, чтобы новые чаты не назначались. Нужны отдельные статусы «Перерыв» и «Обед» с отображением времени за текущий день; контрольные лимиты — 15 минут и 1 час соответственно.
- Полноценная оценка качества и расширенные отчёты нужны, но не блокируют первый тест.

# План доработок

## P0 — до запуска теста

| Доработка | Критерий готовности |
|---|---|
| Автоматический разбор очереди ожидания | Если обращение ожидает из-за полного лимита или завершения обработки ботом, то при освобождении подходящего оператора оно назначается автоматически, без ручного действия администратора. |
| Равномерная маршрутизация | Новые и ожидающие обращения распределяются между доступными операторами равномерно и с учётом лимитов одновременных чатов. |
| Статусы и контроль доступности | «Завершает диалоги» исключает новые назначения; добавлены «Перерыв» и «Обед». На панели видны статус и длительность за текущий день. Онлайн задаётся только вручную. |
| Стабильность базового контура | Бот отвечает по сценарию; оценка клиента и её отображение в интерфейсе не требуют обновления страницы. Причины инцидентов фиксируются и устраняются. |

## P1 — первая волна после стабилизации

| Доработка | Критерий готовности |
|---|---|
| Карточка клиента | Телефон приходит из интеграции либо редактируется вручную. При технической возможности отображаются гео и платформа iOS/Android. |
| Внутренние комментарии | Комментарии видны в ленте диалога отдельным стилем и не уходят клиенту. |
| Тематики | Оператор вводит часть названия и быстро выбирает тематику из отфильтрованного списка. |
| Обратная связь клиента | После любой числовой оценки есть необязательное поле комментария; ответ не создаёт новый диалог и виден в истории обращения. |
| Панель операционного контроля | Для каждого оператора видны статус и число текущих чатов; доступны активные/ожидающие диалоги и отбор по онлайн-операторам. |
| История клиента | Из результата поиска по телефону, имени или каналу открывается история всех обращений с датой и содержимым. |
| Шаблоны | Реализован поиск и автоподбор в поле чата, вставка выбранного шаблона по Enter, поиск и редактирование в справочнике шаблонов. |

## P2 — развитие после первого теста

| Доработка | Критерий готовности |
|---|---|
| Отчёты | Реализован согласованный ежедневный отчёт и выгрузка по переданному ТЗ. |
| Оценка качества | Настраиваемая форма аудита с нужными параметрами и комментарием к каждому параметру. В минимальном отчёте — оператор, число проверок и оценки по каждому проверенному диалогу. |
| Расширенная аналитика качества | После запуска базовой формы — агрегаты и графики, состав которых определяется отдельно. |
| Эскалация и «спасение» | Уточнить и развить передачу на вторую линию/в общую очередь и сценарий «спасение» с таймером и уведомлением. Сейчас не критично. |
| Бот и база знаний | Спроектировать импорт и поддержку знаний на основе текущего процесса в ChatBoost; проверить сценарии, каналы, промпт и правила передачи оператору. |
| Роли и настройки | Убрать отдельную роль старшего сотрудника, если её права полностью совпадают с администратором; уточнить необходимость отдельных линий. |

## Приёмка перед тестом

1. Смоделировать заполненные лимиты операторов, перевод обращения в ожидание и появление свободного слота.
2. Проверить равномерность распределения серии новых и ожидающих обращений между несколькими операторами.
3. Проверить статусы «Завершает диалоги», «Перерыв» и «Обед»: отсутствие новых назначений и корректный учёт времени.
4. Проверить полный базовый путь: бот → оператор → тематика → причина закрытия → цифровая оценка и комментарий клиента.
5. Провести короткий UAT с Екатериной по панели, карточке клиента и шаблонам после реализации P1.
