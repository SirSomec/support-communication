# План доработок: управление ботом, знания и полноценная консультация (итерация 2 программы BAI)

> **Статус:** утверждённый план доработок.
> **Дата:** 13 июля 2026.
> **Продолжает:** [продуктовый план AI-ботов](2026-07-12-bots-ai-agent-product-plan.md), [execution guide BAI](2026-07-12-bots-ai-agent-execution-guide.md), [ADR BAI-001](2026-07-12-bai-001-scenario-ai-contract-adr.md).
> **Причина:** первая итерация программы BAI дала работающий фундамент (runtime, публикация, AI-ответ по знаниям, безопасность), но продукт остаётся неинтуитивным: настройка и знания разбросаны по разделам, тест — одно сообщение вместо чата, рамки ответов не настраиваются, интерфейсы документов/ссылок/MCP неполны.

## 2. Зафиксированные продуктовые решения (13.07.2026)

- [x] **«Дообучение» = пополнение знаний + обратная связь.** Бот улучшается через новые документы/статьи/URL/MCP, разбор «вопросов без ответа» и ревью оценок операторов. Fine-tuning модели провайдера в скоуп не входит (подтверждено владельцем).
- [x] **Новый раздел «Знания» в главной навигации.** Единый хаб: статьи, документы, страницы (URL), MCP-подключения, «вопросы без ответа». Статьи переезжают из «Качества».
- [x] **MCP: тенант добавляет — администратор сервиса одобряет.** Заявка от администратора организации, активация после approve; read-only ограничения сохраняются.
- [x] **Тест-чат — только живой.** Песочница всегда вызывает реальный AI и реальный retrieval; без готового AI-подключения тест недоступен (заглушка с причиной и CTA). Расход токенов виден и лимитируется. Локальная проверка триггера в мастере остаётся как есть.

Ориентиры индустрии: Intercom Fin (единый knowledge hub, живой preview-чат, unresolved questions), Zendesk AI agents (procedures/guidance, поэтапная готовность контента).

## 5. Фаза I — Живой тест-чат и консультационный цикл (P0)

Самая острая жалоба. Чат делает осмысленным всё остальное (рамки, знания, версии).

- [x] **BAI-800 — Консультационный цикл в runtime.** Узел `ai_reply` получает режим «консультация»: после ответа сценарий остаётся в узле и обрабатывает следующие реплики клиента тем же узлом (обновляя compact session state), пока не сработает выход: клиент получил ответ и попрощался, запросил человека, исчерпан `maxClarifyingQuestions`/`maxTurns`, нарушение policy, ошибка провайдера → handoff. Конфиг узла: `maxTurns`, `clarifyingQuestionsLimit`, существующие `fallbackMessage/handoffQueue`. Обратная совместимость: старые версии сценариев работают по-прежнему (один проход).
- [x] **BAI-801 — Sandbox-сессии.** Реализовано на `DurableStore` (JsonFileStore + inMemory, как agent-session-state): сессии эфемерны (TTL 2 ч), Prisma-таблица не требуется; изоляция достигается ephemeral in-memory `AutomationRepository` на каждый ход. Сущность `BotSandboxSession` (tenant, scenario, revision/versionId, канал, locale, автор, TTL) + изолированный runtime-инстанс `kind="sandbox"`: side effects не покидают песочницу — `message_delivery` возвращается в ответе, `bot_handoff` показывается как событие чата, webhook-узлы по умолчанию не вызываются (заглушка с пометкой в trace, переключатель «выполнять webhooks» для админа). Prisma + JSON parity, аддитивная миграция.
- [x] **BAI-802 — API песочницы.** `POST /automation/bot-scenarios/{id}/sandbox-sessions`, `POST …/sandbox-sessions/{sid}/messages`, `DELETE …/sandbox-sessions/{sid}`. Ответ на реплику: сообщения бота, события (handoff, собранное поле), trace: сработавший триггер/правило, путь по узлам, retrieval-фрагменты со score и версиями источников, вызов MCP (после Фазы L), guardrail-решения, tokens input/output/cached, стоимость, latency. Реальный вызов LLM — всегда (решение №4). Идемпотентность реплик по `messageId`.
- [x] **BAI-803 — Бюджет теста.** Лимит `limits.sandboxMonthlyTokenBudget` подключения либо `BOT_SANDBOX_MONTHLY_TOKEN_BUDGET` (по умолчанию 100k); расход дополнительно записывается в общий месячный бюджет подключения. Расход песочницы учитывается в месячном бюджете подключения и отдельном лимите `limits.sandboxMonthlyTokenBudget`; в UI — расход текущей сессии и остаток; при исчерпании — понятная ошибка с CTA к service-admin. Метрики usage помечаются `context="sandbox"`.
- [x] **BAI-804 — UI чата.** Смонтирован на текущем экране «Боты» вместо песочницы-textarea; перенос во вкладку «Тест» консоли — BAI-810. Полноценный чат: пузыри клиент/бот, «бот печатает…», системные события (передача оператору, сбор контакта), сворачиваемая панель «Как бот думал» (trace по каждой реплике), выбор канала и языка, «Начать заново», выбор «черновик / опубликованная версия». Без готового AI-подключения — заглушка: причина человеческим языком + ссылка на настройку (по ролям). Первично монтируется на текущем экране «Боты» вместо песочницы-textarea, затем становится вкладкой «Тест» консоли (BAI-810).
- [~] **BAI-805 — Регрессионные кейсы.** Кнопка «Сохранить как проверку» и запись `AutomationBotTestRun` готовы; автопрогон сохранённых кейсов в publish-checklist объединяется с red-team (BAI-844). Кнопка «Сохранить диалог как проверочный»: реплики уходят в `AutomationBotTestRun.cases`; в publish-checklist появляется пункт «Регрессия пройдена» с запуском сохранённых кейсов (живой прогон, бюджет теста).
- [x] **BAI-806 — Тесты.** `backend/tests/bai-806-bot-sandbox-contracts.test.ts` (13 кейсов) + обновлённый браузерный смок; попутно исправлена регрессия: нормализация версий затирала `triggerRules: undefined` в `[]`, из-за чего фразовые триггеры опубликованных версий без снапшота правил не срабатывали. Контракты: изоляция песочницы (ни одного outbound side effect), идемпотентная реплика, cross-tenant отказ, недоступный провайдер → корректная ошибка чата, бюджет-стоп, консультационный цикл (уточнение → ответ → handoff по лимиту), TTL-очистка сессий.

**Критерий фазы:** администратор ведёт с ботом многоходовой диалог с настоящими AI-ответами и видит по каждой реплике, почему бот ответил именно так и сколько это стоило.

## 6. Фаза J — Консоль сценария и прозрачность (P0)

- [x] **BAI-810 — Страница сценария с вкладками** `Обзор · Настройка · Знания · Тест · Версии · Результаты` (реализует раздел 11.2 продуктового плана). «Боты» превращается в список + вкладку «Проактивные приглашения»; выбор сценария открывает консоль.
- [x] **BAI-811 — Вкладка «Обзор»: паспорт сценария.** Человеческим языком: где работает (каналы), когда запускается (фразы/режим/приоритет), как отвечает (правила/AI, тон, язык), на каких знаниях (источники со статусами готовности), когда передаёт человеку (условия + очередь), готовность AI, последняя публикация и последний тест. Все предупреждения readiness — с кнопкой «Исправить», ведущей в нужную вкладку/раздел.
- [x] **BAI-812 — Вкладка «Настройка»: карточки шагов + черновик поверх публикации.** Черновик хранится как overlay `BotScenario.draft` (аддитивная миграция); runtime исполняет закреплённую версию, sandbox draft-режим тестирует overlay, publish материализует и очищает его; `discard-draft` откатывает. Канал ноды в advanced-редакторе — select по каналам сценария. Редактирование в форме карточек (как мастер, но для существующего сценария); canvas/JSON — только внутри «Расширенных настроек». Добавить «Создать черновик изменений» для опубликованного сценария (draft-ревизия следующей версии по ADR BAI-001, §1.3) — сейчас правка требует остановки бота, это главный источник ощущения «непрозрачно и страшно трогать». В редакторе ноды поле «Каналы» заменить на нормальный мультиселект.
- [x] **BAI-813 — Вкладка «Версии».** Diff-сводка по полям + `POST /bot-scenarios/{id}/rollback` с аудитом. Список версий с автором/датой/статусом, человекочитаемый diff (что изменилось: триггеры, источники, промпт, policy), «Откатиться к версии» (backend `rollbackToPublishedVersion` уже есть) с подтверждением и записью в аудит.
- [~] **BAI-814 — Вкладка «Результаты».** Операционная панель перенесена во вкладку; новые агрегаты (вопросы без ответа сценария, оценки) появятся вместе с Фазами K/N (BAI-826/852). Перенос `ScenarioOperationalPanel` + агрегаты: доля ответов с источником, handoff по причинам, вопросы без ответа этого сценария, AI usage/cost/latency, оценки операторов. Закрывает остатки BAI-507 (видимость trace/usage по ролям, retention).
- [x] **BAI-815 — Чистка экрана «Боты».** Удалены селекторы «Боты по каналам» и выдуманная карточка «Нерабочее время»; метрики и аудит вынесены во вкладку «Работа ботов»; textarea-песочница заменена чатом ещё в Фазе I. Реальное расписание нерабочего времени не выдумываем — отдельная задача вне итерации. Удалить селекторы «Боты по каналам» (противоречат детерминированному выбору по триггерам/приоритету — оставить бейджи каналов на карточках), статичную карточку «Нерабочее время» заменить настоящим расписанием тенанта (используется триггером «В нерабочее время») либо перенести в настройки канала; убрать песочницу-textarea (заменена чатом).
- [~] **BAI-816 — A11y/responsive/локализация** консоли: aria-атрибуты, сегментированные вкладки и брейкпоинты покрыты (responsive-смок 390/768/1024/1440 зелёный); полный клавиатурный/скринридер-аудит — в Фазе O (BAI-862).

**Критерий фазы:** новый администратор открывает сценарий и за минуту понимает, когда бот запускается, что и на чём отвечает, когда зовёт человека, что изменилось в последней версии и как безопасно внести правку.

## 7. Фаза K — Раздел «Знания»: документы, страницы, «дообучение» ч.1 (P0)

- [ ] **BAI-820 — Раздел «Знания» в навигации.** `navigationModel.js` + `access.js` (права `knowledge.read/write`), экран на `ProductScreen` с вкладками «Статьи · Документы · Страницы · MCP-подключения · Вопросы без ответа». Метрики сверху: источников готово/с ошибками, статей опубликовано, вопросов без ответа за 7 дней.
- [ ] **BAI-821 — Переезд статей.** `KnowledgeBaseWorkspace` перемещается во вкладку «Статьи»; в «Качестве» остаются оценки/QA/AI-скоринг, на старом месте — подсказка-ссылка. Governance-цикл и вложения не меняются.
- [ ] **BAI-822 — Менеджер источников.** Единая таблица всех источников (тип, статус ingest/index, approval, readiness, версия, число чанков, последняя индексация, «используется в N сценариях»), действия: обновить/переиндексировать, одобрить, отключить, архивировать, удалить; фильтры по типу/статусу, поиск. API-дополнения: `PATCH /knowledge-sources/{id}` (название, расписание), `DELETE /knowledge-sources/{id}` (запрет, если привязан к published-сценарию — список блокирующих сценариев в ошибке), `GET /knowledge-sources/{id}/usage`.
- [ ] **BAI-823 — Документы-файлы как источники.** Загрузка PDF/DOCX/TXT/MD/HTML через существующий безопасный контур upload + ClamAV (`WorkspaceFile`), извлечение текста, chunking, статусная лента этапов «загружен → антивирус → извлечение → индексация → готов/ошибка» с человеческими кодами ошибок. Ограничения типа/размера видимы до загрузки.
- [ ] **BAI-824 — Страницы (URL) по-взрослому.** Расписание синхронизации, отметка «содержимое изменилось» (diff checksum), предпросмотр извлечённого текста, видимая пользователю allowlist-политика и понятные отказы SSRF-защиты («страница на приватном адресе — запрещено»).
- [ ] **BAI-825 — Прозрачность индексации.** Просмотр чанков источника («что именно знает бот») + «Проверить поиск»: строка запроса → найденные фрагменты со score, тем же retrieval API, что использует бот.
- [ ] **BAI-826 — «Вопросы без ответа».** Копить события «retrieval пуст/ниже порога» и «handoff из-за отсутствия знаний» (причины уже пишутся в step journal / agent_run): список с текстом вопроса (PII-редакция), частотой, сценарием/каналом; действия «Создать статью» (prefill вопросом) и «Скрыть». Это первая половина «дообучения».
- [ ] **BAI-827 — Связка статей и источников.** Публикация новой версии статьи автоматически обновляет document-источник (есть `refresh-document`) и показывает это в UI; в статье бейдж «Используется ботами: N».
- [ ] **BAI-828 — Миграции и тесты.** Аддитивные миграции (usage-связи, unanswered-очередь), Prisma+JSON parity, негативные тесты удаления/cross-tenant, обновление playwright-стабов и смоков.

**Критерий фазы:** администратор в одном разделе видит всё, что знает бот, загружает документ или страницу, проверяет поиск по ним и превращает «вопрос без ответа» в статью.

## 8. Фаза L — MCP: полный цикл с одобрением (P1)

- [ ] **BAI-830 — Расширение модели `McpConnector`.** Поля: `name`, `description`, `authType` (`none|bearer|custom_header`), `secretCiphertext/keyVersion` (через существующий `SecretStore`), `requestedBy/requestedAt`, статусы `draft → pending_approval → approved → enabled ↔ disabled → archived`; изменение endpoint/tools сбрасывает approval (поведение сохраняется). Секрет не возвращается из API.
- [ ] **BAI-831 — Tenant UI + API (вкладка «MCP-подключения»).** Создать заявку (endpoint, авторизация, назначение), видеть статус («на одобрении», «одобрено», «отклонено: причина»), после одобрения — каталог инструментов (`tools/list`), выбор разрешённых read-tools, health-check «Проверить», включить/выключить, лимиты видимы. Права `knowledge.write`.
- [ ] **BAI-832 — Service-admin UI одобрений.** Очередь заявок MCP (endpoint, хост против глобального allowlist, запрошенные tools), одобрить/отклонить с причиной, управление глобальным host-allowlist из UI (сейчас env `MCP_CONNECTOR_ALLOWED_HOSTS`), immutable-аудит решений.
- [ ] **BAI-833 — MCP в retrieval.** `knowledge-retrieval.service` для `mcp`-источников вызывает `McpReadOnlyConnectorService.invoke` (разрешённый read-tool, нормализованный вопрос как аргумент) с таймаутом/лимитами; результат становится passages с citation «MCP: имя/tool»; ошибка/таймаут = отсутствие доказательств → уточнение или handoff, не выдумывание. Короткий TTL-кэш результата (ключ без PII), инвалидация при disable.
- [ ] **BAI-834 — MCP в trace.** Вызовы MCP видны в тест-чате (tool, длительность, усечённый результат) и в handoff-карточке оператора (источник «MCP»).
- [ ] **BAI-835 — Безопасность.** Сохранить write-regex-блок, allowlist, input/result caps; негативные тесты: write-tool, приватный IP, оверсайз, отозванное одобрение посреди диалога, секрет не в логах/аудите.

**Критерий фазы:** организация сама подключает свой read-only MCP-сервер, сервис-админ одобряет за пару кликов, бот отвечает по данным MCP с видимой citation, и всё это не открывает write/SSRF.

## 9. Фаза M — Рамки консультации (guardrails) (P0, после Фазы I)

- [ ] **BAI-840 — Модель `agent_policy`.** На сценарий (+ наследуемые tenant-дефолты): запрещённые темы (список с формулировками и примерами), темы «только оператор», обязательность источника (вкл. по умолчанию), `maxClarifyingQuestions`, максимальная длина ответа, тон/язык (переносятся из конфига узла), редактируемый шаблон вежливого отказа, дополнительные «правила поведения» (bounded-текст, попадает в system prompt после безопасностных rails и не может их переопределить). Версионируется вместе со сценарием (`BotScenarioVersion`). Закрывает остатки BAI-500.
- [ ] **BAI-841 — UI «Рамки ответов».** Карточка в вкладке «Настройка»: «О чём бот отвечает / о чём не отвечает / когда сразу зовёт человека», без технических терминов, с предпросмотром «как бот откажет». Изменения policy — через черновик и публикацию, с аудитом.
- [ ] **BAI-842 — Policy evaluator в runtime.** Pre-check входящего вопроса (v1 — нормализованные списки тем/стоп-слов; v2 — дешёвый классификатор-роутер) и post-check ответа (стоп-темы, длина, наличие citations при фактическом ответе). Нарушение → шаблон отказа или handoff `reason="policy"`; каждое решение — в policy trace `agent_run`.
- [ ] **BAI-843 — Пороги уверенности.** Настраиваемый retrieval score threshold (строгий по умолчанию) и опциональный self-check достаточности доказательств перед ответом; ниже порога — уточняющий вопрос (в пределах лимита) или handoff. Закрывает остатки BAI-504 вместе с quota/rate-limит guardrail.
- [ ] **BAI-844 — Red-team перед публикацией.** Обязательный набор кейсов: prompt injection из документа/URL/MCP, запрос персональных данных, запрещённая тема, «позови человека», недоступный провайдер. Прогоняется в песочнице (живой), провал блокирует публикацию в checklist с объяснением.
- [ ] **BAI-845 — Видимость guardrails.** В тест-чате: «ответ ограничен правилом …»; оператору в handoff-карточке: «передано по правилу …»; в «Результатах» — счётчики срабатываний правил.
- [ ] **BAI-846 — Тесты.** Инструкции из документа не переопределяют policy; policy в версии снапшотится; cross-tenant изоляция; изменение tenant-дефолтов не ломает published-версии.

**Критерий фазы:** администратор словами задаёт, о чём бот отвечает, о чём молчит и когда зовёт человека; каждое срабатывание правила видно в тесте, оператору и в результатах.

## 10. Фаза N — Качество консультации и «дообучение» ч.2 (P1)

- [ ] **BAI-850 — Семантический retrieval.** Embeddings через `AiConnection.embeddingModel` (capability уже в контракте): pgvector в Postgres (аддитивная миграция), эмбеддинги чанков при индексации, гибридный поиск (лексика + вектор) со score threshold; JSON-режим без Postgres остаётся на лексике (parity-тест). Закрывает «semantic retrieval» из BAI-501/503.
- [ ] **BAI-851 — Экономный контекст.** Стабильный префикс промпта (policy → state → passages → вопрос), `prompt_cache_key` = tenant+scenarioVersion+policyVersion, метрики cached_tokens/hit-rate в usage. Закрывает остатки BAI-501.
- [ ] **BAI-852 — Ревью обратной связи.** Очередь в «Знаниях»: оценки «Не помогло»/«Неверный источник» с контекстом (вопрос, ответ, citations) → действия «Исправить статью» (prefill), «Отвязать источник от сценария», «Добавить тему в запрещённые». Автоматических правок знаний нет — принцип review сохраняется.
- [ ] **BAI-853 — Статья из диалога.** Из закрытого диалога/handoff — черновик статьи с PII-очисткой, дальше обычный governance-цикл. Кнопка в операторском диалоге и в «Вопросах без ответа».
- [ ] **BAI-854 — Авто-эвалы.** Регрессионные наборы сценария (из BAI-805) + метрики groundedness/citation coverage; автопрогон при изменении знаний/промпта/policy; отчёт во вкладке «Результаты».
- [ ] **BAI-855 — Долги первой итерации.** Связать AI-бюджет с общим quota/tariff ledger (остаток BAI-304); агрегаты usage и retention для trace (остаток BAI-507).

**Критерий фазы:** ответы бота опираются на семантически найденные фрагменты, а «не помогло» от оператора за несколько кликов превращается в исправленную статью — это и есть согласованное «дообучение».

## 13. Метрики успеха итерации

- Время от пустого тенанта до первого опубликованного сценария (цель: < 30 минут без документации).
- Доля сценариев, протестированных в чате до публикации (цель: 100% — enforced checklist'ом).
- Доля AI-ответов с citations; handoff по причине «нет знаний» ↓ после разбора «вопросов без ответа».
- Конверсия «вопрос без ответа → опубликованная статья»; время реакции на «Не помогло».
- Срабатывания guardrails видимы (0 «тихих» отказов); red-team-набор зелёный у 100% публикаций.
- Retrieval hit-rate до/после embeddings; cached_tokens hit-rate; стоимость на решённый диалог не растёт.

## 14. Риски и ограничения

- **Живой тест жжёт токены.** Митигция: отдельный sandbox-бюджет (BAI-803), видимый расход, дешёвая модель подключения — выбор service-admin.
- **pgvector недоступен в JSON-режиме стендов.** Митигция: гибрид с лексическим fallback и parity-тесты (BAI-850); смоки не требуют вектора.
- **Переезд статей ломает привычку.** Митигция: ссылка-редирект из «Качества», объявление в release notes (BAI-821).
- **MCP-секреты и SSRF.** Митигция: существующий SecretStore/allowlist/read-only-regex + негативные тесты (BAI-835); фича за флагом `ai_bot_mcp_sources`.
- **Консультационный цикл увеличивает стоимость диалога.** Митигция: лимиты turns/уточнений (BAI-800), бюджеты и router-pre-check (BAI-842), метрики стоимости на диалог (13).
- **Объём работ.** Фазы I/J/K дают видимый результат независимо; каждая фаза заканчивается работающим срезом за feature flag.

## 15. Открытые вопросы (не блокируют старт)

1. Векторное хранилище: pgvector в основном Postgres или отдельный индекс — решить ADR-ом в BAI-850 по нагрузке пилота.
2. Классификатор запрещённых тем v2 (дешёвый LLM-роутер): включать ли в M или перенести в N — решить по результатам лексической v1.
3. Расписание «Нерабочее время»: отдельная настройка тенанта или канала — уточнить при проектировании BAI-815.
