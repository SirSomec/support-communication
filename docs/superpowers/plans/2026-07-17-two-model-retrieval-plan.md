# План: двухмодельный AI-поиск по знаниям + LLM-бот (AITunnel, кеширование)

Дата: 2026-07-17. Статус: реализуется; P0-P2 выполнены, P3 в работе (2026-07-17).
Связь с итерацией 2: реализует BAI-850 (семантический retrieval) в варианте «LLM-поиск с кешированием» вместо отложенного pgvector; опирается на готовые BAI-851 (prompt cache), BAI-843 (пороги), BAI-825 («Проверить поиск»).

## 1. Цель и результат

Клиентский вопрос ищется по источникам знаний **дорогой** LLM-моделью (полный корпус знаний сценария в кешируемом префиксе промпта — платим за него один раз в TTL, дальше читаем из кеша за ~0.1x), а ответ клиенту составляет **недорогая** модель, как сейчас. Обе модели работают через одно AITunnel-подключение (OpenAI-совместимый API, один ключ, один бюджет). Текущая лексическая логика остаётся нетронутым fallback-путём: смоки, стаб и тенанты без флага работают байт-в-байт как сегодня.

Результат приёмки: администратор в настройках подключения указывает «модель для поиска» (дорогая) и «модель для ответов» (дешёвая); в тест-чате бот находит ответ на перефразированный вопрос («вернуть деньги» → статья «возврат средств»), в trace видны выбранные фрагменты, режим поиска, cached_tokens и стоимость; при недоступности дорогой модели поиск молча деградирует в лексику; citations и «вопросы без ответа» работают без изменений.

## 2. Что уже есть (не переделываем)

- `AiConnection`: `baseUrl`, `chatModel`, `embeddingModel`, `capabilities ["chat_completion","embeddings","retrieval"]`, лимиты, SecretStore — `apps/api-gateway/src/ai-connections/`.
- Транспорт `openai-compatible-chat.provider.ts`: ретраи, таймауты, безопасные ошибки, парсинг `cached_tokens`, `prompt_cache_key` (BAI-851).
- `KnowledgeRetrievalService` (`knowledge-sources/knowledge-retrieval.service.ts`): единая точка поиска для бота, копилота, sandbox-trace и «Проверить поиск»; кэш результатов 5 мин с `purgeSource`-инвалидацией; контракт `KnowledgeRetrievalResult` (passages + citations с offsets).
- Чанки ingestion со стабильными id/offsets (`document-ingestion.ts`), статусная модель источников (ready/approved).
- Учёт расходов `usage.reserve/recordUsage`, месячные бюджеты, rollout-флаги (`evaluateAiAgentsRollout`), policy сценария (BAI-840) с «максимальной длиной ответа».

## 3. Факты AITunnel (docs.aitunnel.ru, проверено 2026-07-17)

- Base URL: `https://api.aitunnel.ru/v1`; ключи `sk-aitunnel-…`; формат OpenAI Chat Completions; endpoint `/v1/embeddings` тоже есть.
- Кеширование (`/features/prompt-caching.html`):
  - `cache_control: {"type":"ephemeral","ttl":"1h"}` — top-level (автоматически) или на content-блоке system-сообщения (явная точка кеша, Anthropic/Gemini/Qwen);
  - OpenAI-модели кешируются автоматически с префикса ≥1024 токенов, запись бесплатна, чтение со скидкой;
  - Anthropic: запись 5 мин ≈1.25x, запись 1 ч = 2x, чтение ≈0.1x; TTL по умолчанию 5 минут, для Anthropic можно `"1h"`;
  - `session_id` (тело или заголовок `x-session-id`, ≤256 символов) — sticky routing, повышает стабильность кеш-хитов у агрегатора;
  - в ответе `usage.prompt_tokens_details.cached_tokens` и `cache_write_tokens`.

## 4. Ключевые решения

1. **Одно подключение, роли моделей — полями.** Аддитивное поле `retrievalModel: string | null` в `AiConnection` рядом с `chatModel`/`embeddingModel`. Двухподключенческий вариант отвергнут: `normalizeCapabilities` принудительно добавляет `chat_completion` каждому подключению, а селектор бота берёт первое ready-подключение по id — второе подключение с дорогой моделью рисковало бы стать ответчиком.
2. **Поиск = LLM-селектор чанков, а не пересказ.** Дорогая модель получает корпус знаний с размеченными id чанков и возвращает строго JSON со списком id (+ `insufficient: true`, если ответа в корпусе нет). Passages собираются из оригинальных чанков по id → дословность «Approved knowledge», citations с offsets и версиями сохраняются без изменений. Пересказ запрещён: он ломает citations и грozит галлюцинациями в знаниях.
3. **Встраивание стратегией внутрь `KnowledgeRetrievalService.retrieve()`.** Контракт `KnowledgeRetrievalResult` не меняется → бот, копилот, sandbox, «Проверить поиск», unanswered questions и 5-минутный кэш работают без правок. Режим выбирается конфигом; любой сбой LLM-пути → текущий лексический код.
4. **Кеш согласно документации AITunnel.** Корпус — отдельный content-блок system-сообщения с `cache_control {type:"ephemeral", ttl:"1h"}`; вопрос клиента — только в user-сообщении; сборка корпуса детерминированная (источники по id, версии в шапке) — иначе кеш не хитуется; `session_id = kr:{tenantId}:{scenarioId}:{corpusChecksum}` (без PII, ≤256); для OpenAI-моделей тот же запрос кешируется автоматически (≥1024 токенов), `prompt_cache_key` уже шлём.
5. **Ответчик остаётся дешёвой моделью.** `chatModel` уже используется ботом; промпт уже собран кеш-дружелюбно (стабильный префикс, знания в конце — BAI-851). Добавляем только `session_id` и опциональный `cache_control` на префиксном блоке + настраиваемый `maxTokens` из policy («максимальная длина ответа») вместо хардкода 500 — закрывает жалобу на обрезанные ответы.
6. **Режим поиска — настройка сценария** (`agent_policy.retrievalMode: "lexical" | "llm"`, дефолт `lexical`), плюс tenant-флаг rollout. Включение — осознанное действие админа; выключение мгновенно возвращает старое поведение.

## 5. Фазы и задачи

### Фаза P0 — транспорт и модель данных (фундамент)

- [x] **BAI-870 — Поле `retrievalModel` в AiConnection.** Аддитивная Prisma-миграция + repository/service/normalize + UI формы подключения: «Модель для ответов (недорогая)», «Модель для поиска знаний (дорогая)», «Модель эмбеддингов (опционально)» с подсказками и примером базового URL AITunnel. Валидация: `retrievalModel` требует capability `retrieval`.
- [x] **BAI-871 — Кеш-примитивы в chat-провайдере.** Расширить `ChatCompletionRequest` аддитивно: `systemBlocks?: Array<{text, cacheControl?: {ttl?: "5m"|"1h"}}>` (сериализуется в content-массив с `cache_control` по докам), top-level `cacheControl?`, `sessionId?` (тело `session_id`). Парсинг `cache_write_tokens` в usage. Существующие вызовы (строковый content) не меняются. Contract-тест на точную форму JSON тела.
- [ ] **BAI-872 — Тестовый фейк.** Расширить фейк-провайдер (http, как в ai-copilot verification) сценарием «селектор чанков»: принимает корпус, возвращает заданные id; фиксирует полученные `cache_control`/`session_id` для асс. Нужен всем фазам ниже.

### Фаза P1 — LLM-поиск (ядро)

- [x] **BAI-873 — Сборка корпуса.** `KnowledgeCorpusBuilder`: по `sourceBindings` берёт ready+approved источники, чанки ingestion (для URL — чанкование `ingestKnowledgeDocument` на лету), собирает детерминированный текст `[c:{sourceId}:{chunkId} v{version}] {content}` с шапкой версий; считает checksum и оценку токенов. Потолок корпуса `RETRIEVAL_CORPUS_MAX_TOKENS` (дефолт 60k): при превышении — предфильтр текущей лексикой до потолка + пометка `corpusTruncated` в результат/trace.
- [x] **BAI-874 — `LlmKnowledgeSearchService`.** Промпт: system-блок 1 — инструкция селектора (вернуть строго JSON `{"chunks":[{"id","confidence"}],"insufficient":bool}`, не отвечать на вопрос, не пересказывать), system-блок 2 — корпус с `cache_control ttl 1h`; user — вопрос клиента (slice 4k, как у бота). Вызов через `retrievalModel`, `responseFormat: json_object`, `temperature 0`, `maxTokens ~300`, `session_id`. Валидация ответа: неизвестные id отбрасываются; пустой список/`insufficient` → пустые passages (существующий путь «нет знаний»); невалидный JSON/ошибка/таймаут → **лексический fallback** с пометкой причины. Отбор в токен-бюджет 1500 и максимум 8 пассажей — как в лексике.
- [x] **BAI-875 — Стратегия в `KnowledgeRetrievalService`.** Параметр `mode` в `retrieve()` (прокидывается из policy сценария; копилот и «Проверить поиск» — следующей фазой). Кэш-ключ дополняется `mode` и checksum корпуса. `recordBotRetrieval` + sandbox-trace: `mode: lexical|llm|llm_fallback`, cached_tokens, латентность поиска.
- [x] **BAI-876 — Бюджет и учёт.** `usage.reserve` перед вызовом селектора (worstCase: первый прогрев = корпус, далее ~вопрос+вывод); `recordUsage` с контекстом `retrieval`; при исчерпании бюджета — лексический fallback, не отказ бота. Sandbox-бюджет учитывает поисковые вызовы.

### Фаза P2 — управление и видимость

- [x] **BAI-877 — Policy/UI сценария.** `agent_policy.retrievalMode` (дефолт lexical) в карточке «Рамки ответов»: «Как бот ищет по знаниям: Быстрый (по словам) / Умный (ИИ, точнее и дороже)» с честной подсказкой про стоимость; версионируется со сценарием, publish-guard не меняется. Tenant-флаг `ai_llm_retrieval` по образцу `evaluateAiAgentsRollout`.
- [x] **BAI-878 — «Проверить поиск» и копилот.** Вкладка «Знания» → «Проверить поиск»: переключатель режима, показ `mode`/fallback-причины/cached_tokens. Копилот оператора получает тот же `mode` из tenant-дефолта policy.
- [x] **BAI-879 — Ответчик: кеш и длина.** `session_id` (= promptCacheKey) и опциональный `cache_control` на стабильном префиксе промпта бота; `maxTokens` ответа из policy «максимальная длина ответа» (дефолт поднять до 1000) вместо хардкода 500. usage-дашборд: cached hit-rate по контекстам (ответы/поиск) — закрывает хвост BAI-851/855.

### Фаза P3 — приёмка

- [ ] **BAI-880 — Контракты.** Селектор возвращает известные id → passages с корректными citations; невалидный JSON → fallback + trace-причина; `insufficient` → unanswered question записан; корпус байт-идентичен между вызовами (кеш-стабильность); превышение потолка → предфильтр + пометка; cross-tenant изоляция; бюджет-стоп → fallback; parity: `mode=lexical` даёт текущие результаты (снапшот-тест).

## 6. Экономика кеша (ориентир для выбора моделей)

Корпус C токенов, вопросов N за TTL-час: лексика — 0; LLM-поиск (Anthropic-класс) ≈ `2×C` (прогрев 1h-кеша) + `N×(0.1×C + вопрос + ~300 вывода)`. При C=30k и N=50/час дорогая модель читает ~3k «эффективных» токенов корпуса на вопрос вместо 30k. OpenAI-модели: прогрев бесплатный, чтение со скидкой, кеш автоматический ≥1024 токенов. Прогрев повторяется при любом изменении корпуса (checksum в session_id делает это видимым в метриках). Дешёвая модель ответчика — без изменений расхода + экономия на кешируемом префиксе.

## 7. Риски и деградации

| Риск | Митигация |
| --- | --- |
| Дорогая модель недоступна/лимит/таймаут | Автоматический лексический fallback, причина в trace; бот никогда не падает из-за поиска |
| Модель возвращает мусорные id / пересказ | Строгий JSON-контракт, валидация id по корпусу, пустой валидный остаток → fallback |
| Кеш не хитуется (нестабильный префикс) | Детерминированная сборка корпуса + тест байт-идентичности; session_id sticky routing; метрика cache_write_tokens алертит на постоянные прогревы |
| Корпус больше потолка | Лексический предфильтр до потолка + `corpusTruncated` в trace; embeddings-предфильтр — отдельная фаза при реальной потребности |
| Рост стоимости незаметен админу | Расход поиска виден в usage (контекст retrieval), sandbox показывает стоимость реплики, месячный бюджет подключения общий |
| PII: вопрос клиента уходит в дорогую модель | Так же, как сегодня в дешёвую; корпус — только одобренные знания; в session_id PII нет |
| Смоки/стаб без сети | Режим по умолчанию lexical; llm-режим только за флагом и с фейк-провайдером в тестах |

## 8. Открытые вопросы (не блокируют P0)

1. Дефолт `ttl` кеша корпуса: 1h (дороже прогрев, реже прогревы) или 5m — решить по частоте вопросов пилота после метрик P1.
2. Embeddings-предфильтр для корпусов >60k токенов (модели в AITunnel есть: text-embedding-3-small и др.) — делать только если появятся такие тенанты.
3. Копилоту оператора дорогой поиск включать сразу или после пилота на боте — решить в P2 по метрикам.
