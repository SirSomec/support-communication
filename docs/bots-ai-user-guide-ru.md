# Руководство: боты и AI-агент (для администратора и поддержки)

Как настроить консультационного бота без кода: источник знаний, ключевая фраза, AI, handoff, удаление/восстановление и диагностика.

## 1. Подготовка AI (Service Admin)

1. Выберите tenant.
2. Создайте AI connection (совместимый OpenAI-like endpoint), задайте budget/RPM. При желании задайте отдельный лимит токенов на тестирование (`sandboxMonthlyTokenBudget`).
3. Пройдите Test connection. Ключ после сохранения не читается — только маска.
4. При инциденте: Disable connection (kill switch уровня провайдера).

## 2. Знания (tenant-admin)

Раздел **«Знания»** в главной навигации собирает всё, что знает бот:

1. **Статьи** — редактор статей базы знаний (черновик → проверка → публикация). Из опубликованной статьи создаётся источник для бота.
2. **Документы** — загрузка файлов (TXT/Markdown/HTML): антивирус → извлечение текста → индексация. Дождитесь статуса «Готов» и одобрите.
3. **Страницы** — HTTPS-URL: сервер безопасно читает страницу (только allowlist; private IP и redirect отклоняются), вы проверяете предпросмотр и одобряете.
4. **MCP-подключения** — заявка на read-only MCP-сервер; администратор сервиса одобряет, после чего подключение можно «Сделать источником».
5. **Вопросы без ответа** — вопросы, на которые у бота не было знаний; одним кликом создаётся черновик статьи.
6. **Обратная связь** — оценки операторов «не помогло»/«неверный источник»; разберите их или создайте исправляющую статью. Знания не меняются автоматически.

У каждого источника: предпросмотр «что знает бот» и «Проверить поиск» (тот же поиск, что у бота). Отключение/архив/удаление запрещены, пока источник привязан к сценарию.

## 3. Ключевая фраза, сценарий и консоль

1. Создайте сценарий в мастере: задача → запуск → как помогает → знания и передача → проверка.
2. Откройте сценарий — это **консоль** с вкладками: **Обзор** (паспорт), **Настройка** (поля + рамки), **Знания**, **Тест** (живой чат), **Версии** (откат), **Результаты**.
3. В настройке задайте ключевые фразы и режим совпадения (содержит / точно / токены), тон, язык, очередь handoff.
4. **Рамки ответов**: о чём бот не отвечает (вежливый отказ), по каким темам сразу зовёт человека, обязателен ли источник, строгость поиска, дополнительные правила поведения. Правила безопасности платформы эти рамки не отменяют.
5. Правки опубликованного сценария копятся в **черновике следующей версии** — клиенты продолжают видеть текущую версию, пока вы не опубликуете изменения.
6. Публикация через checklist. Пауза не удаляет опубликованную версию; версию можно откатить на вкладке «Версии».

## 3a. Живой тест-чат

1. Вкладка **«Тест»** (или кнопка «Тест-чат») — полноценный чат: вы пишете как клиент, бот отвечает по-настоящему (реальный запуск, поиск, AI).
2. Панель «Как бот думал» показывает сработавший триггер, найденные фрагменты, вызов AI, токены и задержку; события (передача оператору, сбор поля, срабатывание рамок) видны в ленте.
3. Реальные диалоги и каналы не затрагиваются; расход теста ограничен отдельным бюджетом токенов.
4. «Сохранить как проверку» превращает диалог в регрессионный набор сценария.

## 4. Handoff и оценка оператора

1. При низкой уверенности, ошибке провайдера или выключенном пилоте клиент получает fallback, диалог уходит оператору.
2. AI-агент может передать диалог и сам: если клиент просит человека или вопрос вне его возможностей, модель завершает ответ служебным маркером `[[HANDOFF]]`. Маркер вырезается до отправки — клиент видит только вежливую фразу о передаче (причина в карточке: «AI-агент передал диалог оператору»). Если свободных операторов нет, диалог встаёт «В очереди» и виден во вкладке «Ожидают».
3. В чате оператора видна карточка: цель, state, AI outcome, citations, причина.
4. Оператор отмечает **Помогло / Не помогло / Неверный источник**. Feedback tenant-scoped и **не меняет** статьи/источники без отдельного review.

## 5. Удаление и восстановление

1. Удаление (архив) требует подтверждения названием.
2. Восстановление возвращает сценарий **выключенным** — проверьте и включите вручную.
3. Legal/audit hold блокирует purge.

## 6. Диагностика ошибок

| Симптом | Куда смотреть | Первое действие |
| --- | --- | --- |
| Бот не стартует | Trigger preview / priority conflict | Уточнить фразу и каналы |
| Нет ответа по знаниям | Раздел «Знания»: статус источника, «Проверить поиск» | Переиндексировать / одобрить / сменить binding |
| Бот отказал или сразу зовёт оператора | Вкладка «Настройка» → «Рамки ответов»; в чате событие «ответ ограничен правилом» | Уточнить запрещённые/операторские темы, порог поиска |
| Частые handoff | Вкладка «Результаты» + fallback reason | Разобрать «вопросы без ответа», обновить источники или pause |
| MCP не отвечает | Раздел «Знания» → MCP: статус (на одобрении/отклонено), health | Дождаться одобрения сервис-админа / проверить allowlist |
| Provider errors | Service Admin connection test, alerts | Kill switch connection / quota |
| Пилот «молчит» для tenant | Flag `ai_agents_v1` allowlist | Добавить tenant или выключить enforce |

Метрики: `bot_*` в workspace telemetry. Алерты и owners: [runbook](bots-ai-operations-runbook.md).

## 7. Эскалация инцидента

1. Соберите `tenantId`, `scenarioId`, `traceId` (без текста диалога и секретов).
2. Tenant-admin → support-lead (качество).
3. Support-lead → service-admin (provider/quota).
4. Platform-ops → security (SSRF/MCP).
5. Коммуникация клиенту: кратко, без технических ключей; статус после kill switch / restore.
