# План разработки фронтенда платформы поддержки и коммуникаций

Версия: 1.0
Дата: 2026-06-26
Статус: рабочий план фронтенд-разработки
Основание: [functional-requirements-support-communication-platform.md](functional-requirements-support-communication-platform.md)

## 1. Цель фронтенда

Собрать полноценный рабочий интерфейс омниканальной платформы поддержки уровня Usedesk/Jivo: операторский cockpit, панель старшего сотрудника, клиентские профили, шаблоны, отчеты, настройки прав, каналов, SDK и расширенные механики качества.

Фронтенд не должен выглядеть как MVP или набор демо-страниц. Каждый раздел проектируется как рабочая поверхность для ежедневного использования: плотная, сканируемая, с ясными состояниями, быстрыми действиями, понятной навигацией и проверяемыми пользовательскими сценариями.

## 2. Текущий стек и ограничения

- React 19 + Vite.
- `lucide-react` для иконок.
- Локальная навигация через состояние `section`; полноценный роутинг пока не введен.
- Основной UI уже построен в `src/App.jsx`, продуктовые разделы вынесены в `src/sections.jsx`, стили в `src/styles.css`.
- Dev server: `http://127.0.0.1:5173/`.
- Дизайн-система уже задана текущим cockpit: темная левая навигация, белые рабочие панели, синие primary actions, компактные таблицы, радиус 8px, без декоративных hero-паттернов.

## 3. Уже реализовано во фронтенде

- Операторский экран `Диалоги`: очередь, выбранный чат, карточка клиента, история, теги, рекомендации, закрытие диалога.
- Внутренний комментарий, который визуально отделен от клиентских сообщений.
- Выбор тематики в чате и запрет закрытия без тематики.
- Раздел `Панель`: операторы онлайн/перерыв/offline, активные диалоги, ожидание, SLA-риски, очереди по каналам.
- Раздел `Клиенты`: таблица профилей с телефоном, каналом, устройством, тематикой и историей.
- Раздел `Шаблоны`: список, поиск, редактор, доступы, переменные, создание шаблона.
- Сохранение шаблона прямо из composer в чате; новый шаблон попадает в рекомендации и на вкладку `Шаблоны`.
- Раздел `Отчеты`: показатели, каналы, дайджест, таблица метрик, экспорт XLSX как UI-сценарий.
- Раздел `Настройки`: роли, каналы, лимиты операторов, критичные правила.
- Подключения и SDK-консоль: SDK, Telegram, MAX, VK, health, snippet, события SDK.
- Исходящий диалог по телефону через `Быстрые действия`: телефон, канал, тематика, стартовое сообщение, создание нового диалога.
- Desktop и mobile responsive-проверки для ключевых новых сценариев.

## 4. Принципы UI/UX

- Первый экран приложения всегда рабочий интерфейс, не landing page.
- Операционные разделы должны быть плотными, но не перегруженными: таблицы, списки, статусы, фильтры, панели действий.
- Основные команды должны быть icon + label, icon-only кнопки обязаны иметь `aria-label`, `title` и понятный hover/focus state.
- Не использовать вложенные карточки внутри карточек; рабочие поверхности строить как панели, таблицы, строки и модальные окна.
- Все критичные действия имеют явный feedback: toast, disabled state, validation text или подтвержденное изменение состояния.
- На mobile допускается вертикальная компоновка, но без потери основных действий и без горизонтального overflow.
- Роли и права должны отражаться в UI: доступность разделов, disabled controls, подписи причин ограничения.
- Все отчеты, фильтры и таблицы должны быть пригодны для сканирования и экспорта.

## 5. Архитектурный план фронтенда

### 5.1. Ближайшая структуризация кода

- Вынести seed-данные из `App.jsx` и `sections.jsx` в `src/data/`.
- Разделить крупные компоненты на feature-модули:
  - `features/dialogs/`
  - `features/templates/`
  - `features/reports/`
  - `features/settings/`
  - `features/integrations/`
  - `features/supervisor-panel/`
- Вынести общие UI-примитивы:
  - `ProductScreen`
  - `SectionTitle`
  - `MetricTile`
  - `ChannelBadge`
  - `Modal`
  - `Toast`
  - `ToolbarSearch`
  - `EntityTable`
- Сохранить текущую визуальную систему без рестайлинга ради рестайлинга.

### 5.2. Состояние и будущая интеграция API

- Пока использовать локальный state для интерактивных сценариев.
- Подготовить слой `services/` с мок-адаптерами под будущий backend:
  - `dialogService`
  - `templateService`
  - `reportService`
  - `integrationService`
  - `permissionService`
- Для каждого сервиса описать UI loading, empty, error, partial states.
- Не привязывать UI напрямую к текущим seed-массивам.

## 6. Фазы разработки

### Фаза 1. Укрепление операторского cockpit

Цель: сделать экран диалога полноценной рабочей поверхностью оператора.

Задачи:
- Добавить реальные состояния очереди: `Мои`, `Ожидают`, `SLA`, `Все`, `Спасение`, `Низкая оценка`.
- Добавить фильтры по каналу, тематике, оператору, SLA, статусу и наличию внутренних комментариев.
- Добавить действия над диалогом: назначить, передать, поставить на паузу, эскалировать, вернуть в очередь.
- Добавить вложения в composer: заглушка загрузки, список файлов, ошибки размера/типа.
- Добавить edit/preview перед отправкой AI-подсказки или шаблона.
- Добавить timeline событий: назначение, передача, смена тематики, закрытие, возврат в очередь.

UI/UX acceptance criteria:
- Оператор видит причину попадания диалога в очередь и следующий ожидаемый шаг.
- Все критичные статусы читаются без открытия карточки клиента.
- Закрытие без тематики невозможно и визуально объяснено.
- Internal note невозможно спутать с клиентским сообщением.

### Фаза 2. Шаблоны и база знаний

Цель: довести контур шаблонов до полноценного инструмента оператора и команды.

Задачи:
- Разделить шаблоны на личные, командные, глобальные.
- Добавить категории, теги, каналы, тематики, переменные, usage analytics.
- Добавить быстрый поиск и вставку шаблонов из composer.
- Добавить создание шаблона из выделенного сообщения и из черновика.
- Добавить preview шаблона с подстановкой переменных.
- Добавить редактор базы знаний: статьи, статусы публикации, каналы, связанные тематики.
- Добавить рекомендации статей по тематике и тексту обращения.

UI/UX acceptance criteria:
- Оператор может сохранить шаблон из чата не покидая диалог.
- Новый шаблон сразу доступен в рекомендациях и на вкладке `Шаблоны`.
- Шаблон показывает область доступа и канал.
- Редактор не перегружает оператора техническими полями.

### Фаза 3. Каналы, SDK и исходящие диалоги

Цель: закрыть полный frontend-контур подключений SDK, Telegram, MAX, VK и исходящих контактов.

Задачи:
- Расширить `Настройки -> Подключения` до отдельных карточек/деталок канала.
- Для каждого канала показать:
  - статус подключения;
  - последнее успешное событие;
  - ошибки интеграции;
  - доступных сотрудников;
  - лимиты;
  - тест отправки/приема;
  - технические идентификаторы.
- Добавить SDK playground:
  - init;
  - identifyUser;
  - initConversation;
  - trackEntryPoint;
  - syncTopic;
  - device payload preview.
- Расширить исходящий диалог:
  - поиск существующего клиента;
  - выбор сценария;
  - проверка доступности канала;
  - предупреждение о дублях;
  - аудит создания.

UI/UX acceptance criteria:
- Администратор понимает, какой канал сломан и почему.
- Старший сотрудник видит только доступные ему действия.
- Исходящий диалог нельзя создать без валидного телефона и стартового сообщения.
- SDK-события имеют понятные человекочитаемые описания.

### Фаза 4. Клиентские профили и история

Цель: сделать карточку клиента полноценным источником контекста.

Задачи:
- Добавить детальную страницу клиента.
- Добавить объединение дублей профилей.
- Показать устройства, ОС, версию приложения, язык, timezone, точку входа.
- Показать историю всех каналов и предыдущих обращений.
- Добавить маскирование чувствительных данных по правам.
- Добавить заметки по клиенту и системные атрибуты.
- Добавить события SDK до начала чата: экран, источник, время в приложении.

UI/UX acceptance criteria:
- Оператор видит номер телефона, устройство и точку входа без поиска в других системах.
- Чувствительные данные скрываются при недостаточных правах.
- История помогает ответить, а не превращается в длинный лог без приоритета.

### Фаза 5. Панель старшего сотрудника

Цель: дать старшему сотруднику управляемую панель смены.

Задачи:
- Разделить виджеты: online, break, offline, active dialogs, waiting, SLA, rescue.
- Добавить перераспределение очередей по лимитам.
- Добавить настройки перерывов и статусов операторов.
- Добавить drill-down в оператора: диалоги, нагрузка, SLA, качество.
- Добавить предупреждения: перегруз, просрочка, низкая оценка, спасение чата.
- Добавить bulk actions: передать группу диалогов, поднять приоритет, назначить старшему.

UI/UX acceptance criteria:
- Состояние смены понятно за 30 секунд.
- Проблемные очереди и операторы визуально выделены.
- Перераспределение имеет preview до применения.

### Фаза 6. Отчеты, дайджест и экспорт

Цель: закрыть управленческую отчетность и выгрузки по всем показателям.

Задачи:
- Добавить фильтры: период, канал, оператор, тематика, команда, статус, SLA.
- Добавить ежедневный отчет и дайджест руководителя.
- Добавить таблицу всех доступных метрик с экспортом.
- Добавить графики:
  - новые обращения;
  - закрытые;
  - первое время ответа;
  - SLA;
  - нагрузка операторов;
  - тематики;
  - CSAT/CSI;
  - спасенные/пропущенные чаты.
- Добавить export queue: XLSX, CSV, PDF.
- Добавить историю выгрузок и audit trail.

UI/UX acceptance criteria:
- Все метрики из интерфейса можно выгрузить.
- Дайджест читается как управленческий summary, а не как набор чисел.
- Экспорт имеет понятный статус: готовится, готов, ошибка.

### Фаза 7. Права, роли и администрирование

Цель: реализовать UI для сотрудника, старшего сотрудника и администратора.

Задачи:
- Добавить role switcher для frontend-проверки сценариев доступа.
- Добавить матрицу прав:
  - видимость панели;
  - общие настройки;
  - смена паролей;
  - отчеты;
  - каналы;
  - шаблоны;
  - база знаний;
  - просмотр чувствительных данных.
- Добавить управление сотрудниками.
- Добавить смену пароля сотруднику старшим сотрудником.
- Добавить настройки доступных каналов на сотрудника.
- Добавить лимит чатов на оператора и исключения.

UI/UX acceptance criteria:
- Пользователь понимает, почему действие недоступно.
- Администратор видит полный доступ, старший сотрудник не видит/не меняет глобальные настройки.
- Изменения прав визуально попадают в аудит.

### Фаза 8. Качество, CSAT/CSI и AI-помощь

Цель: добавить контур контроля качества и AI-инструментов без подмены оператора.

Задачи:
- Добавить CSAT/CSI после закрытия диалога.
- Добавить фильтр низких оценок.
- Добавить карточку оценки: балл, комментарий клиента, оператор, тематика, канал.
- Добавить ручную оценку качества ответа старшим сотрудником.
- Добавить AI-панель:
  - summary диалога;
  - suggested reply;
  - suggested topic;
  - tone/risk detection;
  - quality check before send;
  - suggested article.
- Добавить явные кнопки: принять, вставить, редактировать, отклонить.

UI/UX acceptance criteria:
- AI-подсказка не отправляется автоматически.
- Все AI-действия маркированы как вспомогательные.
- Старший сотрудник видит качество по оператору, каналу и тематике.

### Фаза 9. Proactive, посетители и спасение чатов

Цель: добавить конкурентные механики Usedesk/Jivo, включенные в требования.

Задачи:
- Добавить раздел активных посетителей/пользователей SDK до начала чата.
- Добавить proactive invitations:
  - условия показа;
  - cooldown;
  - тексты по сегментам;
  - preview;
  - A/B test;
  - статистика принятия.
- Добавить режим спасения чата:
  - таймер после принятия;
  - предупреждение оператору;
  - возврат в очередь;
  - уведомление старшему;
  - отчет по спасенным и пропущенным чатам.
- Добавить privacy controls для наблюдения за контекстом и печатью.

UI/UX acceptance criteria:
- Proactive-настройки понятны не только разработчику.
- Режим наблюдения отделен от очереди обращений.
- Таймер спасения виден, но не перегружает чат.

### Фаза 10. Полировка, доступность и production readiness

Цель: довести интерфейс до уровня стабильного продукта.

Задачи:
- Ввести системный responsive QA для 390, 768, 1024, 1440 px.
- Проверить keyboard navigation и focus states.
- Проверить `aria-label`, `aria-modal`, role dialog, tables, buttons.
- Ввести empty/loading/error states для всех экранов.
- Добавить visual regression checklist.
- Разделить CSS на feature-файлы или ввести CSS modules, если файл станет плохо поддерживаемым.
- Добавить smoke/e2e сценарии для критичных flows.
- Подготовить Storybook или внутреннюю страницу UI-kit, если компонентная база продолжит расти.

UI/UX acceptance criteria:
- Нет горизонтального overflow на mobile.
- Нет clipped text в кнопках, таблицах, карточках и модалках.
- Нет неактивных controls без объяснения.
- Все модальные окна закрываются предсказуемо и не конфликтуют с одинаковыми labels.

## 9. QA-gates для каждой frontend-итерации

- `npm run build` проходит.
- Browser/IAB smoke-test: page identity, not blank, no framework overlay, console без `error/warn`.
- Проверен основной interaction path новой функциональности.
- Проверены desktop и mobile viewport.
- Проверен screenshot через `view_image` против принятой дизайн-системы.
- Нет горизонтального overflow.
- Нет clipped text в основных controls.
- Icon-only controls имеют `aria-label`, `title` и понятную область клика.
- Новые модальные окна имеют `role="dialog"` и `aria-modal="true"`.
- Новые empty/error/loading states описаны или явно отложены в backlog.

## 10. Definition of Done для frontend-фичи

Фича считается готовой, если:

- пользовательский сценарий можно пройти без обращения к коду;
- изменения состояния видны в UI, а не только в toast;
- есть validation и disabled states для некорректных действий;
- desktop и mobile не ломают компоновку;
- новая поверхность соответствует текущей дизайн-системе;
- данные и компоненты не создают дублирующую логику, если уже есть общий state;
- есть понятный путь будущего подключения API;
- изменение зафиксировано в git отдельным коммитом.
