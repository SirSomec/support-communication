# План: рабочий продукт «Боты и AI-автоматизации»

> Статус: `черновик плана`  
> Владелец продукта: команда Support Communication  
> Обновлено: 2026-07-12

## 1. Цель и принятые решения

Превратить текущий раздел «Боты» из редактора статичных flow в работающий no-code продукт для консультаций клиентов: бот отвечает только на основании разрешённых источников, объяснимо передаёт диалог человеку и не выполняет действий в CRM клиента.

### Зафиксированные решения

- [x] Подход: гибридный. Детерминированные правила нужны для запуска, сбора данных, маршрутизации и handoff; AI нужен для консультационных ответов по источникам.
- [x] Первая продуктовая область: консультационные вопросы. Изменение заказов, возвраты, платежи и любые записи в CRM не входят в этот релиз.
- [x] Источники: внутренние статьи, документы, URL и MCP-серверы.
- [x] Настройка AI-провайдера и ключа выполняется только Администратором сервиса, отдельно для каждого tenant.
- [x] Tenant-администратор настраивает знания, сценарии, каналы и тесты, но не видит API-ключи провайдера.

### Что не входит в первый релиз

- [ ] Изменение данных в CRM клиента и другие write-действия через API/MCP.
- [ ] Голосовой бот, исходящие кампании и генерация изображений.
- [ ] Обучение модели на пользовательских диалогах.

## 2. Definition of Done

План считается завершённым только когда одновременно выполняются все условия:

- [ ] Администратор сервиса в UI выбирает tenant, подключает или ротирует ключ совместимого AI-провайдера, выбирает модели ответов и embedding, устанавливает лимиты и успешно запускает проверку соединения.
- [ ] Секрет нельзя прочитать после сохранения ни через API, ни через UI, ни в логах; доступны только маска, дата проверки и история ротаций.
- [ ] Tenant-администратор без кода создаёт, редактирует, тестирует, публикует, выключает, архивирует и при допустимых условиях удаляет сценарий.
- [ ] Триггер «ключевая фраза» содержит реальные фразы и правила совпадения; runtime сопоставляет входящий текст и фиксирует причину запуска.
- [ ] Клиент получает полезный ответ только при наличии релевантных разрешённых источников; интерфейс и журнал показывают, какие источники использованы.
- [ ] При отсутствии источника, низкой уверенности, запросе человека или ошибке провайдера бот корректно уточняет вопрос либо передаёт диалог оператору по настроенному правилу.
- [ ] Документы, URL и одобренные read-only MCP-источники можно подключить, проверить, ограничить tenant/каналом/языком, переиндексировать и отключить.
- [ ] Для каждого ответа записаны версия сценария, модель, источник, причина handoff, задержка, входные/выходные/cached токены и стоимость без хранения API-ключа.
- [ ] Сценарий проходит интерактивный тест и регрессионный набор до публикации; опубликованную версию можно откатить.
- [ ] На pilot-tenant достигнуты согласованные SLO, безопасность и продуктовые метрики из раздела 13.

## 3. Целевой пользовательский путь

```text
Список сценариев
  → Создать сценарий
  → Когда запускать?
  → Как помочь клиенту?
  → Откуда брать ответ?
  → Когда передать человеку?
  → Протестировать в чате
  → Опубликовать версию
  → Смотреть результаты и улучшать знания
```

### Информационная архитектура

| Раздел | Результат для пользователя |
|---|---|
| `Сценарии` | Видит, что работает, на каких каналах, с каким источником и когда проверялось. |
| `Сценарий` | Настраивает поведение обычными словами, без JSON и canvas по умолчанию. |
| `Знания` | Подключает и одобряет статьи, документы, URL и MCP-источники. |
| `Тест` | Общается с ботом как клиент и видит путь, источники и причину handoff. |
| `Результаты` | Понимает, где бот помог, не нашёл знания или передал человеку. |
| `AI и подключения` | Администратор сервиса безопасно подключает провайдера для отдельного tenant. |

### Основные UX-правила

- [ ] Не использовать в первом слое слова `node`, `canvas`, `schema`, `webhook`, `JSON`, `RAG` и `handoff`; заменить их понятными названиями и короткими объяснениями.
- [ ] Оставить расширенный flow-редактор и импорт/экспорт только в режиме «Расширенные настройки».
- [ ] Всегда показывать статус AI: `не настроен`, `готов`, `ограничен лимитом`, `ошибка подключения`.
- [ ] Не позволять публиковать сценарий с пустыми источниками, конфликтующими триггерами, отключённым AI в AI-режиме или отсутствующей очередью передачи.
- [ ] Во всех опасных действиях объяснять последствие и предоставлять безопасную альтернативу: выключить, архивировать, удалить.

## 4. Этап 0 — продуктовый контракт и baseline

### 4.1. Зафиксировать доменную модель

- [ ] Описать единый `Scenario` с полями: tenant, каналы, приоритет, статус, режим ответа, источники, правила запуска, правила передачи, версии и владельцы.
- [ ] Ввести статусы `draft`, `published`, `disabled`, `archived` и правила переходов между ними.
- [ ] Описать, что публикация создаёт неизменяемую версию, а активный диалог закрепляется за её версией до завершения.
- [ ] Определить минимальный набор типов шагов: фиксированный ответ, спросить данные, быстрые варианты, условие, ответ по знаниям, передача оператору, завершение.
- [ ] Зафиксировать, что read-only MCP — источник консультации, а не действие. Любая попытка вызвать write-инструмент должна блокироваться политикой.

### 4.2. Согласовать измеримые целевые показатели

- [ ] Утвердить baseline: доля консультационных диалогов, время первого ответа, доля handoff, CSAT, темы без ответа.
- [ ] Утвердить pilot-цели: доля ответов с источником, доля корректных handoff, потолок стоимости на диалог, p95 latency.
- [ ] Утвердить список безопасных тестовых тем и red-team сценариев: попытки получить персональные данные, prompt injection из документа/URL/MCP, пустые источники, провайдер недоступен.

### 4.3. Технический аудит и миграция

- [ ] Составить карту существующих `automation`, `quality`, `knowledge`, `conversation`, `service-admin` модулей и используемых JSON/Prisma хранилищ.
- [ ] Создать миграционный план от текущих `flowNodes` к новой модели, не ломая существующие опубликованные сценарии.
- [ ] Добавить feature flag `ai_agents_v1` на tenant и возможность мгновенного выключения AI runtime.
- [ ] Добавить API-контракты и схемы валидации до реализации UI.

**Критерий этапа:** команда согласовала сущности, права, нецелевые функции, метрики и план безопасной миграции.

- [ ] Добавить soft-delete с `deletedAt`, `deletedBy`, причиной и периодом восстановления; не удалять опубликованную историю и audit.
- [ ] Запретить permanent delete, пока сценарий назначен каналу, имеет активную версию или открытые runtime-инстансы.
- [ ] Добавить атомарное снятие назначения с канала перед архивированием/удалением.
- [ ] Записывать immutable audit события всех изменений состояния и назначений каналов.

### 5.2. UI библиотеки сценариев

- [ ] Заменить общий dashboard на список сценариев с поиском и фильтрами: статус, канал, режим ответа, владелец, источник, дата изменения.
- [ ] В карточке показать цель, активные каналы, режим (`Правила`/`AI по знаниям`), очередь передачи, последнюю проверку и живую версию.
- [ ] Добавить меню `⋯`: дублировать, выключить, архивировать, удалить черновик, посмотреть историю версий.
- [ ] Реализовать отдельный диалог удаления: показать затрагиваемые каналы, требовать подтверждение имени и предложить архив вместо удаления.
- [ ] В пустом состоянии объяснить первый шаг: «Подключите знания» или «Создайте сценарий с фиксированным ответом».

### 5.3. Доступность и локализация

- [ ] Добавить видимые focus-state, семантические labels, клавиатурный доступ к меню и диалогам подтверждения.

- [ ] Вынести все пользовательские строки из технических моделей и покрыть русскую локализацию.

**Критерий этапа:** пользователю доступны все жизненные операции со сценарием без кода и без риска повредить опубликованную версию.

## 6. Этап 2 — реальные триггеры и детерминированный runtime

### 6.1. Триггер «Ключевая фраза в сообщении»

- [ ] Добавить набор положительных фраз, вариантов написания и исключающих фраз.
- [ ] Добавить режимы: `любая фраза`, `все фразы`, `точное совпадение`, `фраза внутри сообщения`.
- [ ] Нормализовать регистр, пробелы и пунктуацию; использовать границы слов там, где это уместно.
- [ ] Поддержать язык и канал триггера.
- [ ] Добавить приоритет сценария и валидатор пересечений; один входящий event должен иметь предсказуемый победивший сценарий.
- [ ] Добавить блок «Примеры»: что сработает, что не сработает, почему.
- [ ] Записать в runtime event точный триггер, фразу и правило, которые привели к запуску.

### 6.2. Другие триггеры первого релиза

- [ ] Первое сообщение в новом диалоге.
- [ ] Нерабочее время по timezone tenant и расписанию канала.
- [ ] Ручной запуск оператором для проверки.
- [ ] Не включать семантический intent-trigger без отдельного AI-процесса, теста и понятного объяснения причин.

### 6.3. Runtime

- [ ] Перестроить выбор сценария: tenant → канал → статус опубликованной версии → trigger-match → приоритет.
- [ ] Сделать обработку входящего текста идемпотентной и защищённой от повторной доставки.
- [ ] Хранить сценарное состояние отдельно от полного transcript: выбранный сценарий/версия, собранные поля, текущий шаг, причина передачи.
- [ ] Добавить структурированные ошибки и повторные попытки только для безопасных операций.
- [ ] Покрыть unit- и integration-тестами совпадения, конфликты, приоритеты и переходы.

**Критерий этапа:** фраза, введённая в мастере, действительно запускает только ожидаемый сценарий в реальном канале.

## 7. Этап 3 — сервисная настройка AI-провайдера по tenant

### 7.1. Модель подключения

- [ ] Ввести сущность `TenantAiProviderConnection`: tenant, provider type, base URL, выбранные модели, encrypted secret reference, region/data policy, статус, квоты, версии и timestamps.
- [ ] Поддержать сначала `OpenAI-compatible` provider adapter; сохранить контракт расширяемым для будущих прямых и облачных провайдеров.
- [ ] Разделить модели: `answer model`, `router model`, `embedding model`; не требовать одинаковую модель для всех задач.
- [ ] Хранить ключи в secret manager или зашифрованном vault с key version; в БД держать только reference и маску.
- [ ] Запретить возврат ключа любому API-клиенту после сохранения.

### 7.2. Интерфейс Администратора сервиса

- [ ] В service-admin выбрать tenant и открыть вкладку `AI и знания`.
- [ ] Добавить форму: тип провайдера, endpoint, API key, модели, регион, режим хранения, дневной/месячный бюджет, rate limits.
- [ ] Добавить кнопку `Проверить подключение`: короткий безопасный запрос, проверка модели, прав, embedding и retrieval при наличии.
- [ ] Отображать только маску ключа, состояние, модель, последнее успешное соединение, ошибку без секрета и audit-историю.
- [ ] Реализовать создание, редактирование, ротацию, отключение и отзыв ключа с подтверждением.
- [ ] Ограничить права: tenant admin не может создать/посмотреть/ротировать ключ.

### 7.3. Security и tenancy

- [ ] Обязательный tenant scope для каждой AI-вызова, retrieval-запроса, source connector и cache key.
- [ ] Шифровать секреты at rest, редактировать PII в observability, не журналировать prompt целиком по умолчанию.
- [ ] Добавить allowlist endpoint-ов для OpenAI-compatible provider.
- [ ] Добавить audit: кто настроил провайдера, тестировал, менял модель, лимит или политику.
- [ ] Подготовить режим `provider disabled`: бот не выдаёт AI-ответ, а использует фиксированный fallback или handoff.

**Критерий этапа:** Администратор сервиса может без разработчика безопасно включить AI для одного tenant, а другие tenant не получают ни ключ, ни доступ к его данным.

## 8. Этап 4 — единая система знаний и источников

### 8.1. Каталог источников

- [ ] Использовать существующие статьи базы знаний как первый тип источника.
- [ ] Добавить загрузку документов: PDF, DOCX, TXT, Markdown, HTML; проверка антивирусом, типа, размера и статуса извлечения текста.
- [ ] Добавить URL-crawler: allowlist доменов, preview, robots/лимиты, расписание синхронизации, diff и ручной re-sync.
- [ ] Добавить MCP source connector: endpoint, авторизация, каталог инструментов, только read-only tools, таймауты, лимиты, тест и health-status.
- [ ] Явно разделить `источник знаний` и `действие`. MCP-tool с возможностью записи не может попасть в консультационный режим.

### 8.2. Индексация и правила доступа

- [ ] Создать provider-neutral retrieval interface: ingest, chunk, embed, search, delete, reindex.
- [ ] Хранить source version, chunk metadata: tenant, язык, канал, visibility, audience, дата, статус и контрольная сумма.
- [ ] Фильтровать retrieval по tenant, сценарию, каналу, языку и audience до передачи контекста модели.
- [ ] Показывать статус индексирования: в очереди, готов, частично готов, ошибка; описывать следующий шаг обычным языком.
- [ ] Версионировать и архивировать источники так, чтобы опубликованный сценарий мог быть воспроизведён для аудита.

### 8.3. Управление качеством знаний

- [ ] Ввести approval workflow для источников и статей: черновик, на проверке, опубликован, отключён, архив.
- [ ] Добавить preview найденных фрагментов до публикации источника.
- [ ] Добавить список «вопросов без ответа» с предложением создать статью или расширить источник.
- [ ] Добавить механизм немедленного отключения подозрительного URL/документа/MCP.
- [ ] Реализовать защиту от prompt injection в содержимом: treat-as-data, фильтрация инструкций, allowlist tool calls, тестовый набор вредоносных фрагментов.

**Критерий этапа:** tenant-администратор подключает разрешённый источник, видит его готовность и может включить его только для нужного сценария/канала.

## 9. Этап 5 — AI-ответы с доказательствами и безопасным fallback

### 9.1. Консультационный AI-режим

- [ ] Добавить шаг `Ответить по знаниям` в мастер сценария.
- [ ] Дать выбрать источники, язык, тон, длину ответа, максимальное число уточняющих вопросов и handoff-политику.
- [ ] Запрашивать retrieval до вызова модели; передавать только отобранные фрагменты, их идентификаторы и метаданные.
- [ ] Требовать источник для фактического ответа; без достаточных доказательств бот не должен додумывать ответ.
- [ ] Показывать клиенту ссылки/названия источников, когда это уместно; оператору — полный trace источников и версий.
- [ ] Ограничить output length и запретить ненужные рассуждения/служебные тексты.

### 9.2. Fallback и передача человеку

- [ ] Настроить условия handoff: пользователь попросил человека, знаний нет, низкая уверенность, рискованная тема, лимит попыток, ошибка provider, превышен бюджет.
- [ ] Передавать оператору краткое резюме, вопрос клиента, выбранный сценарий, использованные источники и причину передачи.
- [ ] Не обещать действия, которых продукт не выполняет; для CRM-вопросов использовать честное объяснение и handoff.
- [ ] Настроить поведение, если очередь недоступна: сообщение о сроке ответа и безопасное сохранение обращения.

### 9.3. Контроль качества

- [ ] Собрать эталонный набор консультационных вопросов по каждому tenant/сценарию.
- [ ] Ввести automated eval: groundedness, корректность цитирования, отказ от выдуманных сведений, корректность handoff, тон и PII.
- [ ] Не допускать публикацию AI-сценария без минимального набора зелёных eval и успешного теста подключений.

**Критерий этапа:** бот консультирует по источникам и либо обосновывает ответ, либо безопасно передаёт разговор человеку.

## 10. Этап 6 — экономия токенов, контекста и задержки

### 10.1. Обязательные принципы

- [ ] Никогда не прикладывать к модели всю базу знаний, полный архив диалогов или данные другого пользователя.
- [ ] Для нового клиента начинать с пустого личного контекста: статические инструкции сценария + текущее сообщение + минимальная retrieval-выборка.
- [ ] Не использовать историю другого пользователя как контекст нового обращения; общими могут быть только одобренные публичные знания и анонимные агрегаты качества.
- [ ] Учитывать, что `previous_response_id` избавляет приложение от ручной пересылки transcript, но не обязательно убирает стоимость прошлых input-токенов: у OpenAI они всё ещё биллингуются в цепочке. [Документация OpenAI о conversation state](https://developers.openai.com/api/docs/guides/conversation-state)

### 10.2. Многоуровневый бюджет одного сообщения

- [ ] Ввести дешёвый router: определить язык, intent, риск и необходимость поиска; не вызывать дорогую модель для фиксированного ответа или явного handoff.
- [ ] Выполнять retrieval только для AI-консультации; ограничивать top-k и суммарный token budget контекста.
- [ ] Для ответа использовать модель и `max_output_tokens`, соответствующие консультационной задаче; не включать reasoning высокой глубины без необходимости.
- [ ] Добавить reranking только когда первый поиск недостаточно точен, а не на каждое сообщение.
- [ ] Прерывать цепочку, если пользователь повторяет одинаковое сообщение: предложить уточнение или handoff вместо нового дорогого вызова.
- [ ] Добавить per-tenant и per-scenario budgets: запросы, input/output tokens, стоимость, latency; при превышении — предсказуемый fallback.

### 10.3. Retrieval и knowledge caches

- [ ] Кэшировать embeddings документов и запросов; повторно индексировать только изменившиеся chunk-ы.
- [ ] Кэшировать retrieval result по ключу `tenant + набор источников + версии + язык + нормализованный запрос + audience`; не включать PII в ключ.
- [ ] Сбрасывать retrieval cache при публикации/архивировании источника или смене прав доступа.
- [ ] Использовать строгий score threshold и небольшой top-k; сохранять метрику «качество упало из-за лимита» для настройки.
- [ ] Реализовать semantic answer cache только для не персонализированных FAQ, где совпадают tenant, policy/scenario version, source set/version, язык и права клиента.
- [ ] Не кэшировать ответы, содержащие персональные данные, сведения о текущем диалоге или результаты MCP.

OpenAI прямо отмечает, что ограничение числа retrieval results уменьшает и токены, и задержку, но требует контроля качества. [File search: limiting results](https://developers.openai.com/api/docs/guides/tools-file-search)

### 10.4. Контекст диалога

- [ ] Хранить полный transcript в собственной conversation storage для аудита и оператора, но не пересылать его целиком в каждую модельную операцию.
- [ ] Вести отдельный `agent state`: intent, собранные поля, последние решения, уже заданные вопросы, активный сценарий/версия, source IDs и причина handoff.
- [ ] Передавать модели: `agent state` + последние релевантные реплики + свежие retrieval chunks, а не весь transcript.
- [ ] При достижении порога контекста выполнять compaction: сохранять структурированное краткое состояние и исключать старые реплики из следующего запроса.
- [ ] Для провайдеров, поддерживающих server-side compaction, использовать её при заранее настроенном token threshold; для остальных — provider-neutral summary/compaction service.
- [ ] Проверять, что compaction не потеряла обязательные факты: язык, запрос клиента, уже обещанные действия, собранные данные, ограничения и источник ответа.

Server-side compaction может переносить нужное состояние в меньшем контексте; OpenAI также рекомендует отбрасывать элементы до последнего compaction item в stateless-цепочке. [Документация OpenAI о compaction](https://developers.openai.com/api/docs/guides/compaction)

### 10.5. Кэш неизменяемых инструкций

- [ ] Сформировать стабильный prefix: политика безопасности tenant, опубликованная версия сценария, формат ответа и tool schemas.
- [ ] Помещать статичный prefix до пользовательских данных; динамические данные клиента и retrieval chunks — после него.
- [ ] Использовать `prompt_cache_key`, содержащий tenant, scenario version и policy version, но не user ID и не PII.
- [ ] Регистрировать `cached_tokens`, cache write tokens, input/output tokens, cache hit rate и стоимость на сценарий.
- [ ] Проводить A/B измерения: экономия cache против cache-write cost и возможного ухудшения latency.

Prompt caching работает именно на совпадающем префиксе: неизменяемые инструкции должны идти в начале, а пользовательские данные — в конце. [Документация OpenAI о prompt caching](https://developers.openai.com/api/docs/guides/prompt-caching)

**Критерий этапа:** dashboard показывает token/cost/latency по tenant и сценарию; нагрузочные тесты доказывают, что рост длины диалога не приводит к линейной пересылке всего transcript.

## 11. Этап 7 — no-code интерфейс сценария

### 11.1. Мастер создания

- [ ] Шаг 1. `Цель`: консультация по теме, ответ на частый вопрос, помощь с ориентированием.
- [ ] Шаг 2. `Запуск`: первое сообщение, ключевые фразы, нерабочее время, ручной тест.
- [ ] Шаг 3. `Как помочь`: фиксированный ответ или ответ по выбранным знаниям.
- [ ] Шаг 4. `Когда позвать человека`: условия handoff и очередь.
- [ ] Шаг 5. `Проверка`: визуальный сценарий в обычном языке, список источников, каналы, пример диалога и предупреждения.
- [ ] Предзаполнить безопасные templates и объяснять каждую настройку в контексте.

### 11.2. Экран сценария

- [ ] Вкладка `Обзор`: где работает, цель, текущее состояние, источники, handoff, последняя проверка и публикация.
- [ ] Вкладка `Настроить`: последовательные карточки шагов с режимом «Расширенные настройки».
- [ ] Вкладка `Тест`: интерактивный чат и trace.
- [ ] Вкладка `Версии`: черновик, активная версия, история, diff и rollback.
- [ ] Вкладка `Результаты`: success/handoff/no-answer/source gap/cost/latency.

### 11.3. Прозрачность AI

- [ ] В каждом AI-шаге показать «Бот использует: …» и число доступных источников.
- [ ] Предупреждать, если источники не опубликованы, индексируются, не соответствуют языку/каналу или недоступны выбранной аудитории.
- [ ] В тесте отображать скрытый от клиента trace: найденные фрагменты, версии, выбранный fallback, токены и задержки.
- [ ] В production-журнале дать оператору объяснение ответа без показа секрета, системного prompt или нерелевантного PII.

**Критерий этапа:** неподготовленный tenant-администратор создаёт работающий консультационный сценарий с AI по источникам без документации разработчика.

## 12. Этап 8 — тестирование, публикация и выпуск

### 12.1. Симулятор

- [ ] Создать chat simulator с выбором канала, языка, источников и тестового профиля без PII.
- [ ] Показывать каждый шаг: trigger, найденные знания, ответ, уточнение, handoff или fallback.
- [ ] Поддержать симуляцию ошибок: пустой поиск, timeout MCP, provider unavailable, превышение бюджета.
- [ ] Сохранять успешный диалог как regression case; сравнивать новый черновик с baseline.

### 12.2. Preflight и publish

- [ ] Проверять права, состояние AI connection, источники, индексацию, match-конфликты, безопасные каналы и очередь handoff.
- [ ] Блокировать publish на критических ошибках и ясно показывать исправление.
- [ ] Создавать immutable published version, audit event и rollback-point.
- [ ] Поддержать canary: включить сценарий для ограниченной доли трафика или выбранного канала/аудитории.

### 12.3. Тестовое покрытие

- [ ] Unit: matching, сценарии, source filters, cache key, budget policy, redaction, prompt construction, fallback.
- [ ] Integration: provider adapters, key vault, ingest, URL, документы, read-only MCP, retrieval, runtime, publish/rollback.
- [ ] E2E: service-admin подключает ключ tenant → tenant создаёт источник/сценарий → клиент получает grounded answer → оператор получает handoff.
- [ ] Security: tenant isolation, prompt injection, secret leakage, SSRF/URL crawler, MCP allowlist, access revocation.
- [ ] Load: latency и cost profile на целевых QPS и длинных диалогах.

## 13. Этап 9 — observability, пилот и эксплуатация

### 13.1. Метрики

- [ ] Product: answer rate, assisted resolution rate, handoff rate по причине, unanswered intent, CSAT, повторные обращения.
- [ ] Knowledge: retrieval hit rate, source coverage, source age, queries without evidence, citations per answer.
- [ ] AI: input/output/cached tokens, cache hit rate, embedding/retrieval cost, cost per resolved conversation, p50/p95 latency, provider errors.
- [ ] Safety: blocked prompt injection, PII redactions, policy violations, denied MCP calls, budget fallbacks.
- [ ] Operations: source ingestion health, index lag, MCP availability, queue handoff latency, version adoption.

### 13.2. Pilot

- [ ] Выбрать один tenant, один канал и 3–5 консультационных тем с качественной базой знаний.
- [ ] Провести baseline до включения и задать лимит стоимости/трафика.
- [ ] Запустить canary, ежедневно смотреть неотвеченные вопросы и ручные проверки.
- [ ] Исправлять сначала источники и процедуры, затем prompt/модель; не расширять трафик при падении groundedness или CSAT.
- [ ] Провести security review и incident drill: утёкший ключ, отключённый provider, вредоносный URL, MCP timeout, неверный tenant filter.

### 13.3. Производственный запуск

- [ ] Описать runbook: подключение tenant, ротация ключа, отключение AI, reindex, rollback, бюджет, инцидент.
- [ ] Добавить alerting на provider health, ошибки, индексирование, рост cost, падение cache hit, рост handoff/no-answer.
- [ ] Подтвердить backup/restore источников, сценариев, версий, audit и secret references.
- [ ] Подготовить поддержку: объяснение статусов для клиента, tenant admin и service admin.

## 14. Риски и обязательные ограничения

- [ ] Не допускать межtenant-утечки через retrieval, cache, MCP sessions, audit или error messages.
- [ ] Не включать remote MCP без allowlist, read-only схемы, OAuth/секретной привязки tenant, tool timeout и аудита.
- [ ] Не передавать в модель документ целиком, если достаточно фрагментов; не давать документу управлять системными инструкциями.
- [ ] Не считать AI-ответ достоверным без источника, если вопрос фактический.
- [ ] Не использовать cache между пользователями для персонализированных ответов.
- [ ] Не скрывать стоимость и факт отключённого AI от tenant-администратора.
- [ ] Не выполнять CRM write-действия в рамках этого релиза.
